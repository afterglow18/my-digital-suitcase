Here's the full chain of fixes that got purchases working:

Package lookup never matched (revenuecat.ts) The code was comparing p.packageType against "$rc_monthly" — but the RC SDK uses enum strings like "MONTHLY" / "ANNUAL" / "LIFETIME". No package was ever found, so every purchase returned 'unavailable' before even hitting StoreKit. → Fixed by checking both the identifier ($rc_monthly) and the packageType enum (MONTHLY) in PACKAGE_IDENTIFIERS.
Errors were completely silent (UpgradeSheet.tsx) When the purchase failed, the button just reset to idle with no message. Users had no idea what went wrong. → Added a red error banner that shows the specific reason.
Button stayed greyed out after failure (UpgradeSheet.tsx) If RevenueCat threw an exception instead of returning a result, it bypassed the if/else chain entirely and status stayed "pending" permanently, leaving the button disabled forever. → Wrapped both handlePurchase and handleRestore in try/catch so any throw resets the button.
VITE_REVENUECAT_IOS_API_KEY wasn't being injected by Codemagic (codemagic.yaml) The key existed in Codemagic but the signing group wasn't referenced in codemagic.yaml, so Vite baked undefined as the API key — RC never initialised. → Added groups: [signing] under environment.
Entitlement ID mismatch (revenuecat.ts) The code checked for entitlement "unlock" but the RC dashboard had it named "My Digital Filing Cabinet Pro". → Updated ENTITLEMENT_ID to match exactly.
Tier reverted to Free immediately after purchase (useEntitlements.ts) After a successful StoreKit purchase, visibilitychange fired (paywall closing counts as a visibility event), which triggered syncFromRevenueCat(). In sandbox, RC's servers lag a few seconds behind StoreKit, so the sync got active = false and immediately overwrote the tier back to 'free'. → Added a 60-second grace window (markRecentPurchase) that blocks any downgrade sync after a purchase, plus a scheduled re-verify at 8 seconds once RC has had time to propagate.
